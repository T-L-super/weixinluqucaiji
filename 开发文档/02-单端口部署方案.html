单端口部署方案

文档类型： 架构更新

创建时间： 2026 年 3 月 18 日 23:04

关键变更： 前后端合并

🎯 核心要求： 整个项目上线时只能用一个端口

✅ 解决方案： 前端静态文件由后端 FastAPI 直接服务，前后端合并部署

📌 端口占用： 仅占用 1 个端口（80/443）

1. 架构对比

❌ 原方案（前后端分离）

  • 前端：Nginx（端口 80）
  • 后端：FastAPI（端口 8000）
  • 数据库：MySQL（端口 3306）
  • Redis：（端口 6379）

总端口数：4 个

✅ 新方案（前后端合并）

  • FastAPI + 静态文件（端口 80）
  • 数据库：MySQL（内网，不暴露）
  • Redis：（内网，不暴露）

总端口数：1 个（仅 80）

6. 优势分析

✅ 单端口部署优势

简化部署： 只需开放一个端口，降低运维复杂度

降低成本： 减少端口占用，适合受限环境

提高安全性： 减少暴露面，降低安全风险

便于管理： 前后端统一版本管理，同步更新

开发友好： 本地开发用 SQLite，无需额外配置

7. 开发流程

前后端协同开发

  1. 前端开发： 独立开发，使用 Vite 热重载（端口 5173）
  2. 后端开发： FastAPI 独立运行（端口 8000）
  3. 本地联调： 前端代理到后端，跨域访问
  4. 构建合并： 前端构建（npm run build），产物复制到后端
  5. 统一打包： Docker 镜像包含前后端完整代码
  6. 单端口部署： 启动后仅暴露 80 端口

8. 技术栈更新总结

组件 技术选型 说明
数据库（开发） SQLite 3 轻量级，无需配置
数据库（生产） MySQL 8.0+ 高性能，支持并发
后端框架 FastAPI + Python 同时服务 API 和静态文件
前端框架 Vue 3 + Vite 构建后合并到后端
任务队列 Redis + RQ 内网访问，不暴露端口
部署方式 Docker（单端口） 仅暴露 80 端口
